# 공통 아키텍처 설계 문서 (Common Architecture)

## 1. 개요

본 프로젝트의 모든 마이크로서비스는 일관된 인프라 패턴과 코드 재사용성을 위해 `common-module`에 정의된 공통 아키텍처 가이드를 따릅니다. 특히 MSA 환경에서 데이터 정합성을 보장하기 위한 핵심 패턴들을
기술합니다.

## 2. 재사용 가능한 Transactional Outbox 패턴

각 마이크로서비스는 독립적인 데이터베이스를 가지지만, 이벤트를 Outbox 테이블에 저장하는 **로직의 흐름(Flow)**은 동일합니다. 이를 효율적으로 관리하기 위해 다음 구조를 채택합니다.

### 2-1. 단일 엔티티 재사용 전략 (Shared Entity)

- **위치**: `common-module`
- **형태**: `@Entity` 및 `@Table(name = "outbox")`가 정의된 단일 클래스.
- **설계 결정 배경 (Engineering Trade-off)**:
    - **이상적인 MSA**: 각 마이크로서비스는 물리적으로 독립된 저장소와 독자적인 코드를 가져야 하지만, 사이드 프로젝트 환경에서 모든 도메인을 별개 프로젝트로 관리하는 것은 오버헤드가 크다고 판단했습니다.
    - **현실적인 선택**: 현재 프로젝트는 **멀티 모듈 구조**를 채택하고 있으며, **단일 데이터베이스**를 사용하고 있습니다. 이에 따라 중복된 엔티티 코드를 완전히 제거(DRY 원칙)하고 생산성을
      높이기 위해 `common-module`에 통합된 단일 `Outbox` 엔티티를 정의하여 모든 모듈에서 재사용하기로 결정했습니다.
- **이점**: 서비스 간 일관된 이벤트 데이터 구조 보장, 유지보수 포인트의 단일화, 보일러플레이트 코드 최소화.

### 2-2. 범용 아웃박스 핸들러 전략 (Generic Outbox Handler)

이벤트마다 개별 핸들러 클래스를 만드는 보일러플레이트 코드를 줄이기 위해, `BaseEvent`를 구독하는 **범용 핸들러** 구조를 지향합니다.

- **작동 원리**:
    1. 도메인 서비스에서 `ApplicationEventPublisher`를 통해 `BaseEvent`를 상속받은 이벤트를 발행합니다.
    2. `common-module` 또는 각 서비스의 `CommonConfig`에 정의된 **범용 리스너**가 이를 가로챕니다.
    3. 리스너는 해당 서비스의 `OutboxRepository`를 사용하여 이벤트를 DB에 저장합니다.
- **이점**: 새로운 도메인 이벤트가 추가되어도 별도의 핸들러 클래스를 만들 필요 없이, `BaseEvent`만 구현하면 자동으로 Outbox 처리가 완료됩니다.

## 3. 도메인 이벤트 설계 원칙

### 3-1. BaseEvent 구조

모든 도메인 이벤트는 `BaseEvent<T>`를 상속받아야 하며, 다음 필드를 필수로 포함합니다.

- **eventId (TSID)**: 시스템 전체에서 유일한 이벤트 식별자.
- **eventType**: 이벤트의 성격 (예: `MEMBER_SIGN_UP`).
- **payload**: 실제 전달될 데이터 객체.
- **occurredAt**: 이벤트 발생 시각.

### 3-2. 이벤트 발행 및 트랜잭션 전파

- **동기 저장**: 도메인 로직과 Outbox 저장은 반드시 **하나의 로컬 트랜잭션**으로 묶여야 합니다. (이벤트 발행 시 별도의 `@Async`를 사용하지 않음)
- **비동기 전송**: Kafka로의 최종 전송은 `@TransactionalEventListener(phase = AFTER_COMMIT)`를 사용하여 DB 커밋 성공 후에만 수행합니다.

## 4. 인프라 공통 설정

### 4-1. Redis 네임스페이스 규칙

서비스 간 Redis 키 충돌을 방지하기 위해 다음 규칙을 준수합니다.

- `{service_name}:{domain}:{identifier}` (예: `auth:refresh_token:user:1`)

### 4-2. 예외 처리 모델

- `common-module`에 정의된 `BusinessException`과 `ErrorCode` 체계를 사용하여, 서비스 간 통신 시 일관된 에러 응답 형식을 유지합니다.

## 5. API 멱등성 보장 전략 (Idempotency)

중복 요청(네트워크 재시도, 더블 클릭 등)이 발생해도 시스템의 상태가 한 번만 변경되도록 보장하기 위해 AOP 기반의 멱등성 메커니즘을 적용합니다.

### 5-1. 분산 락 및 캐싱 아키텍처

DB의 Unique 제약 조건 대신 **Redis 분산 락(Distributed Lock)과 응답 캐싱(Response Caching)** 조합을 채택하여 고성능 멱등성을 구현했습니다.

1. **Idempotency-Key**: 요청 헤더에 포함된 고유 키를 기반으로 멱등성을 식별합니다.
2. **Double-Checking 패턴**:
    - 락 획득 전후로 캐시를 이중 확인하여, 선행 스레드가 로직을 수행 중이거나 완료했을 경우 후행 스레드가 비즈니스 로직을 중복 실행하지 않도록 보호합니다.
3. **결과 캐싱**: 비즈니스 로직의 응답(`ResponseEntity`)을 Redis에 저장하여, 이후 동일 요청 시 캐시된 결과를 즉시 반환(O(1))합니다.

### 5-2. 설계 결정 이유 (Trade-off)

- **성능 및 효율성**: 메모리 기반 Redis를 활용하여 DB I/O 부하를 줄이고 중복 요청을 최외곽에서 즉시 차단(Early Exit)합니다.
- **관심사 분리**: 비즈니스 DB 스키마와 멱등성 관리 로직을 분리하여 시스템 복잡도를 낮췄습니다.
- **UX 일관성**: 이미 처리된 요청에 대해 오류 대신 기존 성공 응답을 재반환함으로써 클라이언트의 예외 처리 부담을 경감했습니다.
